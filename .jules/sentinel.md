## 2024-05-24 - Arbitrary file deletion via hardcoded temporary directories
**Vulnerability:** `get-heatmaps.py` hardcoded a temporary directory path (`path_img/'train'`) and subsequently deleted it via `shutil.rmtree` at the end of the script. This created an arbitrary file deletion vulnerability if the user already had a directory named `train` within the provided image path, which would be destroyed without warning. Furthermore, the cleanup code was not wrapped in a `finally` block, meaning any exception during heatmap generation would leave files trapped inside the dummy directory.
**Learning:** Hardcoding temporary directory names in user-provided paths is dangerous, especially when combined with recursive deletion tools like `rmtree`. This pattern assumes full ownership of the filesystem namespace and neglects the possibility of collisions with user data.
**Prevention:** Always use dynamic temporary directory provisioning (e.g., `tempfile.mkdtemp`) to guarantee unique paths and prevent namespace collisions. Additionally, always wrap operations that modify user state or files in `try...finally` blocks to ensure deterministic cleanup and safe restoration of user state, even in the event of errors or exceptions.

## 2025-02-14 - Fix FastAPI File Upload DoS & Information Leakage
**Vulnerability:** The FastAPI endpoints `/predict/image` and `/predict/colors` did not validate file sizes before reading them entirely into memory via `await file.read()`, creating a high-risk Denial of Service (DoS) vulnerability. Additionally, generic `except Exception as e` blocks raised HTTPExceptions that exposed raw exception strings (`str(e)`), risking information leakage about the server's filesystem paths or internal logic.
**Learning:** Upload boundaries aren't natively enforced by FastAPI's `UploadFile` without explicit configuration or checks. Also, attackers can bypass simplistic file size checks by omitting the `Content-Length` header (causing `file.size` to be `None`), so robust checks must explicitly handle `file.size is None`.
**Prevention:** Always define an explicit `MAX_FILE_SIZE` and enforce `if file.size is None or file.size > MAX_FILE_SIZE:` prior to loading content. Ensure error handlers throw sanitized, generic messages instead of returning raw exception strings.
